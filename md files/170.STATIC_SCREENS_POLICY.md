# ?? IMPORTANT: Static Screens Policy

## ?? Screens That Should NOT Be Affected by Themes

The following screens must remain **static** and **unaffected** by the dynamic theme system:

### 1. **Splash Screen** (`SplashScreen.xaml`)
- ? **NO theme integration**
- ? Uses **hardcoded colors**
- ? Logo image display
- ? Rainbow gradient progress bar (static)
- ? Green text (`#00FF00` - hardcoded)
- ? Black background (`#CC000000` - semi-transparent)

**Reason**: Splash screen appears **before** the theme system initializes. Must be self-contained and load instantly.

---

### 2. **Login Window** (`LoginWindow.xaml`)
- ? **NO theme integration**
- ? Uses **hardcoded colors**
- ? Black background
- ? Custom button styles (local resources)
- ? Independent of `App.xaml` merged dictionaries

**Reason**: Login screen is the **first user interaction** and should maintain consistent branding regardless of user preferences. Themes apply **after** successful login.

---

## ?? How Themes Are Applied

### Application Flow

```
1. App Starts
   ?
2. SplashScreen.xaml (STATIC - No themes)
   ?
3. LoginWindow.xaml (STATIC - No themes)
   ?
4. User logs in successfully
   ?
5. MainWindow.xaml loads
   ?
6. ThemeManager.ApplyColorTheme() called
   ?
7. User can select theme in Settings
   ?
8. All subsequent windows use dynamic themes
```

## ?? Technical Details

### Why These Screens Are Static

#### Splash Screen
```csharp
// SplashScreen.xaml.cs
public SplashScreen()
{
    InitializeComponent();
    // Loads BEFORE App.xaml resources are initialized
    // Cannot access DynamicResource from App.xaml
    // Must be self-contained
}
```

#### Login Window
```csharp
// LoginWindow.xaml.cs
public LoginWindow()
{
    InitializeComponent();
    // Opens BEFORE theme preferences are loaded
    // User hasn't logged in yet - no theme data available
    // Maintains consistent branding
}
```

## ?? Best Practices

### Adding New Static Screens

If you need to create a new screen that should remain static (unaffected by themes):

1. **Do NOT** include `App.xaml` merged dictionaries
2. **Do NOT** use `{DynamicResource}` bindings
3. **Use hardcoded colors** or local `Window.Resources`
4. **Test before theme initialization** to ensure independence

### Adding New Themed Screens

If you need to create a new screen that should use themes:

1. ? Use `{DynamicResource AccentBrush}` for colors
2. ? Use `{DynamicResource AccentColor}` for glow effects
3. ? Reference theme-aware styles
4. ? Test with all 12 themes

## ?? Common Pitfalls to Avoid

### ? DON'T Do This (Splash/Login)
```xaml
<!-- BAD: Uses dynamic resources -->
<TextBlock Foreground="{DynamicResource AccentBrush}"/>
<Border BorderBrush="{DynamicResource AccentBrush}"/>
```

### ? DO This Instead (Splash/Login)
```xaml
<!-- GOOD: Hardcoded colors -->
<TextBlock Foreground="#00FF00"/>
<Border BorderBrush="#00FF00"/>
```

### ? DON'T Do This (Splash/Login)
```xaml
<!-- BAD: Merging App.xaml themes -->
<Window.Resources>
    <ResourceDictionary>
        <ResourceDictionary.MergedDictionaries>
            <ResourceDictionary Source="Themes/GreenTheme.xaml"/>
        </ResourceDictionary.MergedDictionaries>
    </ResourceDictionary>
</Window.Resources>
```

### ? DO This Instead (Splash/Login)
```xaml
<!-- GOOD: Local resources only -->
<Window.Resources>
    <Style x:Key="LocalStyle" TargetType="Button">
        <Setter Property="Background" Value="Green"/>
    </Style>
</Window.Resources>
```

## ?? Color Consistency

### Static Color Palette (Splash/Login)
To maintain branding consistency, these screens use:

| Element | Color | Hex Code | Purpose |
|---------|-------|----------|---------|
| Background | Black | `#000000` | Primary background |
| Accent/Border | Neon Green | `#00FF00` | Brand identity |
| Text | Neon Green | `#00FF00` | Readability |
| Overlay | Semi-transparent Black | `#CC000000` | Depth |
| Progress Rainbow | Gradient | Static gradient | Visual appeal |

### Why Green?
The **Matrix Green** (`#00FF00`) is hardcoded in static screens because it's the **brand identity** of The Fallen Collective. This ensures:
- ? Consistent first impression
- ? Recognizable branding
- ? Professional appearance
- ? No theme conflicts

## ?? Resource Loading Order

### Application Startup Sequence

```
1. Windows OS loads executable
   ?
2. App.xaml.cs - Application_Startup() begins
   ?
3. SplashScreen.xaml loads (STATIC)
   ?? Uses only local resources
   ?? NO App.xaml merged dictionaries available
   ?
4. SplashScreen shows loading animation
   ?
5. LoginWindow.xaml loads (STATIC)
   ?? Uses only local resources
   ?? Theme not yet initialized
   ?
6. User logs in
   ?
7. ThemeManager initializes
   ?
8. User preferences loaded (including theme choice)
   ?
9. Theme applied to App.xaml resources
   ?
10. MainWindow.xaml loads (DYNAMIC)
    ?? Uses DynamicResource bindings
    ?? Reflects user's theme choice
```
